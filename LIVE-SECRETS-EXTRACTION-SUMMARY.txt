PURPOSE:
  Extract ALL secrets currently stored in your running Kubernetes cluster,
  including manually-created secrets not in the bootstrap secrets.yaml file.

FILES CREATED:
  1. extract-live-secrets.sh      (5.6 KB) - Bash script version
  2. extract-live-secrets.py      (9.8 KB) - Python script version
  3. LIVE-SECRETS-EXTRACTION-GUIDE.md    - Complete guide
  4. LIVE-SECRETS-EXTRACTION-SUMMARY.txt - This file

EXTRACTION INCLUDES:
  ✓ All Kubernetes Secrets (from all namespaces)
  ✓ All ConfigMaps (often contain sensitive config)
  ✓ Database credentials (PostgreSQL, Redis, etc)
  ✓ API keys and tokens (Nextcloud, Authentik, etc)
  ✓ TLS certificates and keys
  ✓ Docker registry credentials
  ✓ Any manually-created secrets via kubectl commands
  ✓ System secrets (kube-system namespace)
  ✓ Automatically generated secrets

WHAT IS EXTRACTED:
  • Secrets Kind: All Secret objects (type: Opaque, TLS, docker-json, etc)
  • ConfigMaps Kind: All ConfigMap objects
  • Data: Base64-encoded (not encrypted, but reversible only with access)
  • Format: Kubernetes API manifest format (kind: List with items)
  • Namespaces: All namespaces (argocd, db, cloud, auth, immich, monitor, backup, etc)

OUTPUT FILE FORMAT:
  secrets-live-backup-YYYYMMDD-HHMMSS.yaml

  Example: secrets-live-backup-20260528-120000.yaml

  Contains:
  - apiVersion: v1
  - kind: List
  - items: all secrets and configmaps
  - Restorable directly with: kubectl apply -f <file>

  BOTH OPTIONS:
  • Automatically discover all namespaces
  • Extract all secrets and configmaps
  • Validate output
  • Print detailed report

SECURING THE BACKUP:

  Encrypt with GPG:
  ─────────────────
  $ gpg --encrypt --recipient your-email@example.com \
        secrets-live-backup-20260528-120000.yaml

  This creates: secrets-live-backup-20260528-120000.yaml.gpg
  Delete the unencrypted file: rm secrets-live-backup-20260528-120000.yaml

  Use git-crypt (if configured):
  ───────────────────────────────
  $ git add secrets-live-backup-*.yaml
  $ git commit -m "Add live secrets backup"
  $ git push  # Encrypted in transit

RESTORATION:

  Restore to new cluster:
  ───────────────────────
  $ kubectl create namespace argocd db cloud auth immich monitor backup
  $ kubectl apply -f secrets-live-backup-20260528-120000.yaml
  $ kubectl get secrets --all-namespaces  # Verify

  Decrypt first (if GPG encrypted):
  ──────────────────────────────────
  $ gpg --decrypt secrets-live-backup-20260528-120000.yaml.gpg > secrets-live-backup-20260528-120000.yaml
  $ kubectl apply -f secrets-live-backup-20260528-120000.yaml

BEFORE SYSTEM REBUILD:

  ✓ Step 1: Run extraction (on your homelab server)
    ./extract-live-secrets.sh

  ✓ Step 2: Encrypt backup (optional but recommended)
    gpg --encrypt --recipient your-email secrets-live-backup-*.yaml
    rm secrets-live-backup-*.yaml

  ✓ Step 3: Store securely
    cp secrets-live-backup-*.yaml.gpg /mnt/nas/backups/
    cp secrets-live-backup-*.yaml.gpg /backup-drive/

  ✓ Step 4: Test restoration (staging cluster if available)
    kubectl apply -f secrets-live-backup-*.yaml
    kubectl get secrets --all-namespaces

  ✓ Step 5: Proceed with rebuild
    ./fresh-install.sh
    # ... run ansible playbooks
    # ... apply restored secrets

TROUBLESHOOTING:

  "kubectl not found"
  ───────────────────
  • Install kubectl on your homelab server
  • Or run scripts with full path to kubectl

  "Permission denied / connection refused"
  ────────────────────────────────────────
  • Ensure you have kubeconfig access
  • Set KUBECONFIG environment variable
  • May need sudo privileges

  "No secrets extracted"
  ──────────────────────
  • Verify kubectl can connect: kubectl get nodes
  • Check namespaces exist: kubectl get namespaces
  • Verify you have permission: kubectl auth can-i get secrets --all-namespaces

FILE SIZES (TYPICAL):

  • 0-5 KB   - Very minimal cluster (few secrets)
  • 5-20 KB  - Typical homelab (several apps)
  • 20-50 KB - Complex setup with many secrets
  • 50+ KB   - Very large number of secrets

SECURITY CONSIDERATIONS:

  ⚠️  Base64 is NOT encryption - it's just encoding
  ✓  Anyone with this file can decode the secrets
  ✓  Store encrypted (GPG, git-crypt, or encrypted storage)
  ✓  Keep file permissions restrictive (chmod 600)
  ✓  Never commit unencrypted to public repos
  ✓  Consider using External Secrets Operator for future deployments

KEY DIFFERENCES FROM BOOTSTRAP:

  secrets.yaml (bootstrap)      │ Live Secrets (current)
  ────────────────────────────────────────────────────
  One-time bootstrap only       │ All secrets in cluster
  Stored on NAS                 │ Extracted from running cluster
  Limited set of secrets        │ Complete inventory
  Used to initialize cluster    │ Used to restore/migrate cluster

EXAMPLE SECRETS THAT WILL BE EXTRACTED:

  From bootstrap:
  • nextcloud-secret (DB_NAME, DB_USER, DB_PASS)
  • wildcard-tls certificate for domain

  Manually created (via kubectl):
  • API tokens added later
  • Database credentials updated
  • OAuth2 client secrets
  • Custom application tokens
  • Additional TLS certificates

  Auto-generated:
  • Service account tokens
  • kubectl credentials
  • Internal cluster certificates

COMPARISON: THREE EXTRACTION METHODS

1. BOOTSTRAP SECRETS (secrets.yaml)
   └─ Location: /mnt/nas/homelab/secrets.yaml
   └─ Tool: ansible/apply-secrets.yml
   └─ Output: secrets-backup.yaml
   └─ Purpose: Initialize new cluster

2. CONFIGURATION INVENTORY (this new method)
   └─ Location: All deployment manifests
   └─ Tool: extract-secrets.py (original)
   └─ Output: secrets-backup.yaml
   └─ Purpose: Track what's needed

3. LIVE EXTRACTION (recommended for rebuild)
   └─ Location: Running Kubernetes cluster
   └─ Tool: extract-live-secrets.sh OR extract-live-secrets.py (NEW!)
   └─ Output: secrets-live-backup-TIMESTAMP.yaml
   └─ Purpose: Backup current state before rebuild

FOR REBUILD: Use Method 3 (Live Extraction)
